# 机器人参数变更规范

## 1. 目标与范围
本规范用于约束 `Core/Inc/Drivers/robot_config*.h` 与 `robot_profile_*.h` 的参数变更流程，确保现场运行参数可追溯、可评审、可回滚。

## 2. 角色与权限（谁可改）
- **参数提交人（开发）**：可提交 `profile_dev`、实验性参数调整。
- **控制算法负责人**：可审批控制参数（PI、滤波、限幅、融合系数）变更。
- **平台/硬件负责人**：可审批板级配置与外设相关参数变更。
- **安全负责人**：必须审批安全阈值、保护策略、超时相关参数变更。
- **发布负责人**：负责将参数合入 `prod` 或车型 profile，并安排版本发布。

> 规则：涉及 `robot_config_safety.h` 或 `robot_profile_prod.h` 的改动，至少需要 **控制/硬件/安全** 中两类负责人共同审批。

## 3. 变更提交流程（如何评审）
1. **新建分支**，修改对应配置文件，并同步更新：
   - `ROBOT_CONFIG_VERSION`（建议格式：`<profile>-YYYY.MM[-rev]`）
   - PR 描述中的“变更前/后参数对照表”与“影响评估”。
2. **本地验证**：
   - 编译通过（目标 profile）。
   - 启动日志确认 profile 名称、版本号、关键阈值摘要正确。
3. **评审要点**：
   - 是否仅在对应文件层级改动（平台/控制/安全/链路/profile）。
   - 是否有越权修改（例如开发直接改 `prod`）。
   - 阈值是否满足安全边界与硬件极限。
4. **合入要求**：
   - 至少 2 人审批，且包含必需角色。
   - 必须关联实车/台架验证记录（可附截图或日志）。

## 4. 回滚策略（如何回滚）
### 4.1 紧急回滚（现场）
- 通过构建参数切回上一稳定 profile，例如：
  - `-DROBOT_PROFILE=prod` 切回标准参数。
  - 或指定上一车型 profile。
- 上电后核对启动日志中的 profile/version/阈值摘要。

### 4.2 代码回滚（仓库）
- 方式 A：`git revert <参数变更提交>` 生成反向提交（推荐，保留审计链）。
- 方式 B：恢复到历史 tag/commit 后重新发布固件。

### 4.3 回滚验收
- 验证项：
  - 编译成功。
  - 启动日志显示为预期回滚 profile 与版本。
  - 安全阈值恢复至基线。

## 5. 命名与版本建议
- Profile 命名：`dev`、`prod`、`vehicle_<id>`。
- 版本建议放在 profile 文件中覆盖 `ROBOT_CONFIG_VERSION`。
- 禁止无版本号参数变更进入发布分支。
